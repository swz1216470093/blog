内存结构

1. 程序计数器

1. 1定义

​ 作用：指向虚拟机下一条执行的指令的地址

​ 特点：

  • 线程私有
  • 唯一一个不会产生内存溢出的区域

2. 虚拟机栈

2.1定义

  • 线程运行时需要的内存空间，称为虚拟机栈

  • 每个栈由栈帧组成，对应着每次方法调用时所占用的内存

  • 每个线程只能有一个活动栈帧，对应着当前执行的方法

    2.2 线程运行诊断

    top命令 查看哪个进程CPU占用过高

    ps H -eo pid,tid,%cpu | grep 进程ID 找到具体是哪个线程占用CPU

    jstack 线程ID | grep nid =十六进制的线程ID

3. 本地方法栈

3.1定义

​ 本地方法运行时需要的内存

4. 堆

4.1 定义

定义：对象存储的空间

4.2 堆诊断

​ jmap -heap pid 查看堆内存占用

​ jmap -dump:format=b,file = a.bin pid 导出内存快照

​ 内存分析工具 分析

5. 方法区

5.3运行时常量池

  • 常量池，就是一张表，存储于.class文件中，虚拟机指令根据这张表找到要执行的类名，方法名，参数类型，字面量（字符串的值，常量的值）信息。
  • 运行时常量池，常量池是*.class文件中的，当该类被加载时，它的常量池的信息就会放入运行时常量池，并把里边的符号地址变为真实地址

5.3.1常量池

5.4字符串常量池

5.4.1 StringTable特性

  • 常量池中的字符串仅仅是符号，第一次用到才变为对象
  • 利用串池的机制，来避免重复创建字符串对象
  • 字符串拼接的原理是StringBuilder（java1.8）
  • 字符串常量拼接的原理是编译期优化
  • 可以使用intern方法，主动将串池中还没有的字符串对象放入串池中
    • 1.8 将这个字符串对象尝试放入串池中，如果有则不放入，没有则放入串池中，然后返回串池中的对象。
    • 1.6 将这个字符串对象尝试放入串池中，如果有则不放入，没有则把此对象复制一份放入串池中，然后返回串池中的对象。

5.4.2 StringTable位置

1.7开始在堆中，因为永久代的回收效率比较低，因为需要Full GC时才会触发回收，而字符串常量池使用非常频繁，如果字符串常量池的回收效率不高，会导致永久代占用大量内存，最终导致永久代OOM

5.4.3 StringTable垃圾回收

5.4.4 StringTable性能调优

StringTable的底层是一个hashtable它的性能和桶大小密切相关

  • 使用-XX:StringTableSize = 桶个数
  • 考虑字符串对象是否入池

6 直接内存

6.1定义

Direct Mermory

  • 常见于NIO操作，用于数据缓冲区
  • 分配回收成本较高，但读写性能高
  • 不受JVM垃圾回收影响

堆内存读写性能不高的原因：

​ java 本身并不具备 IO 读写能力，因此 read 方法调用后，要从 java 程序的用户态切换至内核态，去调用操作系统（Kernel）的读能力，将数据读入系统缓冲区 然后从内核态切换到用户态将系统缓存区的数据拷贝到java缓冲区。

直接内存省略了一步数据拷贝

6.2 直接内存释放原理

  • 使用了 Unsafe 对象完成直接内存的分配回收，并且回收需要主动调用 freeMemory 方法
  • ByteBuffer 的实现类内部，使用了 Cleaner （虚引用）来监测 ByteBuffer 对象，一旦ByteBuffer 对象被垃圾回收，那么就会由一个 叫ReferenceHandler的守护线程通过 Cleaner 的 clean 方法调用 freeMemory 来释放直接内存

6.3禁用显示垃圾回收对直接内存的影响

当使用-XX:+DisableExplicitGC禁用显式GC(System.gc())后，必须等待ByteBuffer对象被垃圾回收掉之后直接内存才能被释放

解决方案：当直接内存使用较多时，直接使用unsafe管理直接内存，使用unsafe.freeMemory释放掉内存

参考：黑马视频《解密JVM》
